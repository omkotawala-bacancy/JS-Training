Variables & Types
The foundation. Before you touch anything else, you have to understand how JavaScript stores data in memory and what the three ways to declare variables actually mean.
JavaScript has 8 data types: 7 primitives (string number bigint boolean null undefined symbol) and 1 structural type (object). Primitives are immutable values stored on the stack. Objects (including arrays & functions) are reference types stored on the heap — your variable holds a pointer, not the value itself.

var is function-scoped and gets hoisted. let and const are block-scoped, live in a "Temporal Dead Zone" before declaration. In modern JS — const by default, let when you need to reassign, never var in new code.
Primitives vs References — the most important distinction
// PRIMITIVES: copied by VALUE
let a = 5;
let b = a;    // b gets a COPY of value 5
b = 99;
console.log(a); // 5 — untouched. a and b are independent.

// OBJECTS: copied by REFERENCE
const user1 = { name: 'Arjun' };
const user2 = user1; // user2 points to the SAME object in memory
user2.name = 'Riya';
console.log(user1.name); // 'Riya' — BOTH changed! Same memory address.

// FIX: clone the object
const user3 = { ...user1 };           // shallow clone
const deep  = structuredClone(user1); // deep clone (nested objects too)
The #1 source of bugs for new devs is mutating an object thinking you're copying it. Always ask: "am I dealing with a primitive or a reference?" Objects need to be cloned before modification.
var vs let vs const — scoping rules
function demo() {
  if (true) {
    var x = 'I leak!';  // scoped to demo(), NOT the if block
    let y = 'Block only'; // scoped to the if block
  }
  console.log(x); // 'I leak!' — var escaped the if block
  console.log(y); // ReferenceError — y is block-scoped
}

// const — can't reassign, but CAN mutate object content
const arr = [1, 2, 3];
arr.push(4); // ✓ fine — you're mutating, not reassigning
arr = [5];   // ✗ TypeError: Assignment to constant variable

Hoisting
Before JavaScript runs a single line, it makes a pass through the file and "hoists" declarations to the top of their scope. Understanding this is crucial for debugging order-dependent bugs.
The JS engine runs in two phases. In the Creation Phase, it scans code and sets up memory for all declarations. var is hoisted and initialized to undefined. Function declarations are fully hoisted — name AND body. let/const are hoisted but placed in the Temporal Dead Zone (TDZ) — they exist but cannot be accessed until their line is reached. In the Execution Phase, code runs line by line with assignments filled in.

Classic interview trap: for(var i=0;i<3;i++) setTimeout(()=>console.log(i),0) prints 3,3,3 — NOT 0,1,2. var is function-scoped and shared across all iterations. When the callbacks fire, the loop is already done and i===3. Fix: use let.
The Event Loop
Arguably the most important concept in JavaScript. Everything async — Promises, DOM events, timers — runs through this system. Understand this and the rest becomes obvious.
JavaScript is single-threaded — it can only do one thing at a time. Yet it handles network requests, timers, and user input without freezing. How? The Event Loop — a system with four components:

1. Call Stack — where code executes. Functions pushed when called, popped when returned. LIFO.
2. Web APIs — when you call setTimeout/fetch/addEventListener, the browser handles the work outside JS on a separate thread.
3. Microtask Queue — higher priority. Promise .then/.catch callbacks, queueMicrotask(), MutationObserver.
4. Callback Queue (Macrotask) — lower priority. setTimeout, setInterval, DOM events.

The rule: After each task, if the Call Stack is empty — drain ALL microtasks first, then take ONE macrotask.

The classic execution order quiz — what prints first?
console.log('1 — sync');            // Call Stack — runs immediately

setTimeout(() => {
  console.log('4 — macrotask');       // Callback Queue — runs last
}, 0);

// Sync code first → ALL microtasks drain → ONE macrotask
setTimeout(fn, 0) does NOT mean "run immediately". It means "put this callback in the macrotask queue after at least 0ms". Sync code and ALL microtasks will always run first. That "0" is a minimum delay, not a guarantee of immediacy.
Asynchronous JavaScript
The evolution from callback hell to elegant async/await. Understanding WHY each pattern exists makes the API click instantly.
JS's async story evolved in three generations.

Generation 1: Callbacks — pass a function to be called when work is done. Simple, but nesting creates "callback hell" — a pyramid of indented code that's nearly impossible to maintain or error-handle properly.

Generation 2: Promises (ES2015) — represent a value that may exist in the future. Chain with .then() to flatten the pyramid. Built-in error handling with .catch().

Generation 3: Async/Await (ES2017) — syntactic sugar over Promises. Makes async code look synchronous. Zero new runtime concepts — it's still Promises under the hood.
Generation 1: Callback Hell — never write this
getUser(1, function(user) {
  getOrders(user.id, function(orders) {
    getProduct(orders[0].productId, function(product) {
      getInventory(product.id, function(inv) {
        // The dreaded pyramid. Error handling per callback. Unreadable.
        console.log(inv);
      }, handleError);
    }, handleError);
  }, handleError);
}, handleError);
Generation 2: Promise Chain — better
getUser(1)
  .then(user    => getOrders(user.id))
  .then(orders  => getProduct(orders[0].productId))
  .then(product => getInventory(product.id))
  .then(inv     => console.log(inv))
  .catch(err   => console.error(err)); // ONE error handler for all!
Generation 3: Async/Await — write this in 2025
async function loadInventory(userId) {
  try {
    const user    = await getUser(userId);
    const orders  = await getOrders(user.id);
    const product = await getProduct(orders[0].productId);
    const inv     = await getInventory(product.id);
    console.log(inv);
  } catch (err) {
    console.error('Something failed:', err);
  }
}
// Reads like synchronous code. try/catch for errors. Clean.
Performance trap — awaiting independent operations sequentially: each await blocks for 1s, total = 3s. If operations don't depend on each other, fire them in parallel with Promise.all.
Sequential (slow) vs Parallel (fast)
// ❌ Sequential — waits for each one before starting the next
async function slow() {
  const a = await fetchA(); // 1s
  const b = await fetchB(); // 1s — total so far: 2s
  const c = await fetchC(); // 1s — total: 3s
}

// ✅ Parallel — all fire at once, total = slowest individual
async function fast() {
  const [a, b, c] = await Promise.all([
    fetchA(), fetchB(), fetchC()
  ]); // total: ~1s
}
Promises
A Promise is a contract: "I don't have this value yet, but I promise to give it to you — or tell you why I couldn't." Three states, immutable once settled.
A Promise has exactly 3 states: pending (initial — neither done nor failed), fulfilled (success, value available), rejected (failed, reason available). Once fulfilled or rejected, a Promise is settled and its state never changes — it's immutable.

Promise combinators: Promise.all([]) — all must succeed, one failure = all fail. Promise.allSettled([]) — waits for all, gives each result regardless of success/failure. Promise.race([]) — settles with the FIRST one to settle (good for timeouts). Promise.any([]) — resolves with FIRST success, only rejects if ALL fail.

Creating a Promise from scratch
const fetchData = new Promise((resolve, reject) => {
  // The executor runs synchronously. Do async work here.
  setTimeout(() => {
    const success = true;

    if (success) {
      resolve({ id: 1, name: 'Arjun' }); // Promise → fulfilled
    } else {
      reject(new Error('Network failed'));   // Promise → rejected
    }
  }, 1000);
});

fetchData
  .then(user    => console.log('Got user:', user))
  .catch(err     => console.error('Failed:', err))
  .finally(()  => hideSpinner()); // always runs — hide loading UI
Promise combinators — when to use each
const p1 = fetch('/api/user');
const p2 = fetch('/api/posts');
const p3 = fetch('/api/settings');

// all: if ONE fails, all fail. For critical required data.
const [user, posts, settings] = await Promise.all([p1, p2, p3]);

// allSettled: shows result of each. For optional/independent data.
const results = await Promise.allSettled([p1, p2, p3]);
results.forEach(r => {
  if (r.status === 'fulfilled') use(r.value);
  else logError(r.reason);
});

// race: first to settle wins. Great for timeout patterns.
const timeout = new Promise((_, rej) => setTimeout(rej, 5000, 'Timeout!'));
const data = await Promise.race([fetch('/api/data'), timeout]);
Always handle promise rejections. An unhandled rejection crashes Node.js processes. In browsers it's a warning today — but treat it like an error. Every Promise chain needs a .catch(), every async function needs a try/catch.
Async / Await
Pure syntactic sugar over Promises. Doesn't add new capability — just makes Promise-based code dramatically easier to read and write.
The async keyword before a function does two things: makes it always return a Promise, and allows use of await inside it. The await keyword pauses execution of the async function (NOT the whole thread!) until the Promise resolves, then gives you the resolved value. Every await is just a .then() callback under the hood. Error handling uses standard try/catch/finally blocks.

Critical mental model: When you await something, the function suspends and control returns to the event loop. Other code can run. When the Promise resolves, the function resumes from where it paused. This is cooperative multitasking — never blocking.
Async/Await deep dive with real fetch pattern
async function getUser(id) {
  try {
    // await pauses HERE until fetch() Promise resolves
    const response = await fetch(`/api/users/${id}`);

    // CRITICAL: fetch() only rejects on network failure, NOT 4xx/5xx!
    // A 404 or 500 response is a "successful" fetch. Check manually:
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }

    // .json() is also async — it reads and parses the response body
    const data = await response.json();
    return data; // becomes the resolved value of getUser()'s Promise

  } catch (err) {
    // catches network errors AND the manually thrown error above
    console.error('Failed to fetch user:', err);
    throw err; // re-throw so callers know it failed

  } finally {
    hideSpinner(); // ALWAYS runs — success OR failure
  }
}

// async function returns a Promise — call it like this:
const user = await getUser(1); // inside another async function
getUser(1).then(u => render(u));  // or chain .then() on the Promise
Go-style error handling — elegant pattern
// Utility: wraps any Promise → returns [error, data] tuple
const to = promise =>
  promise.then(data => [null, data]).catch(err => [err, null]);

async function loadDashboard() {
  const [userErr, user] = await to(getUser(1));
  if (userErr) return showError(userErr); // handle immediately

  const [postsErr, posts] = await to(getPosts(user.id));
  if (postsErr) return showPartial(user); // graceful degradation

  render({ user, posts });
}
// No try/catch nesting. Errors handled at each step.
Don't use await inside Array.forEach — forEach doesn't understand async, the awaits are ignored and your loop finishes before any promise resolves. Use for...of, or Promise.all(array.map(async item => ...)).
The DOM
The Document Object Model is a tree-shaped representation of your HTML in memory. JavaScript can read and modify every node, attribute, and style in real time.
When a browser loads HTML, it parses it and builds a tree of Node objects in memory — the DOM. JavaScript accesses this tree through the document object. Changes to the DOM can trigger the browser to reflow (recalculate layout) and repaint (redraw pixels), which is expensive. Good DOM work means batching writes, using document fragments, and delegating events rather than attaching listeners to every element.

Event bubbling: when you click a button inside a div inside body, the click event fires on the button first, then bubbles up to div, then to body. This is the mechanism behind event delegation.
Selecting elements — modern patterns
// querySelector — CSS selector syntax, returns FIRST match (or null)
const header = document.querySelector('#header');
const btn    = document.querySelector('.btn.primary');
const input  = document.querySelector('input[type="email"]');

// querySelectorAll — returns NodeList (iterable like array, but not Array)
const cards = document.querySelectorAll('.card');
[...cards].forEach(card => card.classList.add('loaded'));

// Tree traversal
header.parentElement;         // go up one level
header.children;              // direct children
header.nextElementSibling;    // sibling after
header.closest('.container'); // walk up to find ancestor matching selector
Creating and modifying elements
const li = document.createElement('li');
li.textContent = 'New item';      // SAFE — escapes HTML automatically
li.innerHTML   = '<b>Bold</b>';   // RISKY — XSS if used with user input!
li.classList.add('item', 'active');
li.setAttribute('data-id', '42');

// Insert into DOM
document.querySelector('ul').append(li);    // after last child
document.querySelector('ul').prepend(li);   // before first child

// PERFORMANCE: batch DOM writes with DocumentFragment
// Appending 100 items one-by-one = 100 reflows. Use a fragment instead:
const frag = document.createDocumentFragment();
items.forEach(item => {
  const el = document.createElement('li');
  el.textContent = item.name;
  frag.append(el);
});
document.querySelector('ul').append(frag); // ONE DOM write, ONE reflow
Event delegation — the professional pattern
// ❌ Beginner: attaches listener to EVERY item
document.querySelectorAll('.list-item').forEach(item => {
  item.addEventListener('click', handleClick);
}); // 100 items = 100 listeners in memory. Breaks with dynamic items.

// ✅ Senior: ONE listener on parent, use event bubbling
document.querySelector('.list').addEventListener('click', (e) => {
  // e.target = the actual element clicked
  // .closest() walks up and finds the matching ancestor
  const item = e.target.closest('.list-item');
  if (!item) return; // clicked on list padding, not an item

  handleClick(item.dataset.id);
});
// ONE listener. Works for dynamically added items too. Memory efficient.
Never use innerHTML with user-provided content. It's an XSS vulnerability — an attacker can inject <script> tags and run malicious code. Always use textContent for text content, or sanitize with DOMPurify if you must render HTML from users.
The this keyword
The most confusing concept in JS. 'this' is NOT about where a function is defined — it's about HOW the function is CALLED. One rule to remember them all.
this is a special variable created for every function execution context. Its value depends entirely on how the function is called, not where it's defined. There are 5 binding rules, checked in order of precedence:

1. new binding — new Foo() → this = newly created empty object.
2. Explicit binding — fn.call(ctx), .apply(ctx), .bind(ctx) → this = what you explicitly pass.
3. Implicit binding — obj.method() → this = obj (the object left of the dot).
4. Default binding — fn() → this = global (window in browser) or undefined in strict mode.
5. Arrow functions — no binding at all. They inherit this lexically from the surrounding scope at the time of definition — not at call time.

The this problem and its arrow function solution
class Timer {
  constructor() { this.seconds = 0; }

  // ❌ Regular function loses 'this' inside callback
  startBroken() {
    setInterval(function() {
      this.seconds++; // 'this' = window/undefined — not Timer!
      // The callback is called as a plain function, not as timer.method()
    }, 1000);
  }

  // ✅ Arrow function inherits 'this' from Timer at definition time
  startFixed() {
    setInterval(() => {
      this.seconds++; // 'this' = the Timer instance ✓
    }, 1000);
  }
}

// call vs apply vs bind
function greet(greeting, punctuation) {
  return `${greeting}, ${this.name}${punctuation}`;
}
const p = { name: 'Riya' };

greet.call(p, 'Hello', '!');        // "Hello, Riya!" — args spread out
greet.apply(p, ['Hello', '!']);    // "Hello, Riya!" — args in array
const boundGreet = greet.bind(p);  // returns NEW function, call later
boundGreet('Namaste', '!');         // "Namaste, Riya!"
Extracting a method from an object loses its context: const { greet } = obj; greet() — 'this' is undefined in strict mode. Fix: const greet = obj.greet.bind(obj) or use class fields with arrow functions: greet = () => { this... }.
map / filter / reduce
The functional trio. Transform data without mutation. Master these three and 80% of your data processing code becomes clean, readable one-liners.
All three are higher-order functions — they accept a function as an argument. They never mutate the original array, always returning a new value.

map(fn) — transforms every element. Input length = output length. Use to convert, reshape, or extract data.
filter(fn) — keeps elements where fn returns truthy. Output is a subset of input.
reduce(fn, initial) — accumulates elements into a single value. That value can be a number, string, object, or array. The most powerful and most flexible of the three — if in doubt, reduce can do what map and filter do.
map — transform every element
const products = [
  { id: 1, name: 'Laptop',   price: 50000, inStock: true  },
  { id: 2, name: 'Mouse',    price: 1500,  inStock: false },
  { id: 3, name: 'Keyboard', price: 3000,  inStock: true  },
];

// Extract just the names — [every element] → [transformed element]
const names = products.map(p => p.name);
// ['Laptop', 'Mouse', 'Keyboard'] — same length, each item transformed

// Apply 18% GST — creates a NEW array, originals UNTOUCHED
const withGST = products.map(p => ({
  ...p,                          // spread existing properties
  price: Math.round(p.price * 1.18),
  originalPrice: p.price
}));
filter — keep elements matching a condition
// Keep only in-stock items
const available = products.filter(p => p.inStock);
// [{ Laptop }, { Keyboard }]

// Chain map + filter — read left to right like English
const availableNames = products
  .filter(p => p.inStock)           // [Laptop, Keyboard]
  .map(p    => p.name.toUpperCase()); // ['LAPTOP', 'KEYBOARD']

// Remove duplicates
const unique = arr.filter((v, i, self) => self.indexOf(v) === i);
// Or cleaner: [...new Set(arr)]
reduce — the Swiss Army knife of array methods
// --- Basic: sum all prices ---
// reduce(callback, initialValue)
// callback(accumulator, currentElement, index, array)
const total = products.reduce((acc, p) => acc + p.price, 0);
// Iteration 1: acc=0,    p=Laptop   → return 0 + 50000 = 50000
// Iteration 2: acc=50000, p=Mouse   → return 50000 + 1500 = 51500
// Iteration 3: acc=51500, p=Keyboard→ return 51500 + 3000 = 54500
// Result: 54500

// --- Advanced: group by property ---
const grouped = products.reduce((acc, p) => {
  const key = p.inStock ? 'inStock' : 'outOfStock';
  (acc[key] ??= []).push(p);
  return acc;
}, {});
// { inStock: [Laptop, Keyboard], outOfStock: [Mouse] }

// --- Advanced: build O(1) lookup map ---
const byId = products.reduce((acc, p) => ({ ...acc, [p.id]: p }), {});
byId[2]; // { id:2, name:'Mouse'... } — instant O(1) access
Build lookup objects with reduce instead of calling .find() inside a loop. find() in a loop = O(n²). A reduce-built lookup map = O(n) to build + O(1) per lookup. For 10,000 items, that's seconds vs milliseconds.
Closures & Scope
A closure is when a function remembers and can access variables from the scope where it was created — even after that outer scope has finished executing. This powers React hooks, modules, memoization, and more.
Every time a function is created in JavaScript, it forms a closure — it carries a reference to its surrounding lexical environment (the scope it was born in). This means the inner function can access outer variables even after the outer function has returned and its call stack frame is gone. The variables live on in memory because the closure holds a reference.

Closures power: data privacy/encapsulation, factory functions, memoization, partial application/currying, and the module pattern. They're also the mechanism behind every React hook you've ever used — useState stores its value in a closure that React manages internally.
Closure basics — a function remembering its birth scope
function makeCounter(initialValue = 0) {
  let count = initialValue; // private — not accessible from outside

  return {
    increment: () => ++count,
    decrement: () => --count,
    reset:     () => { count = initialValue; },
    getCount:  () => count
  };
}

const counter = makeCounter(10);
counter.increment(); // count = 11
counter.increment(); // count = 12
counter.decrement(); // count = 11
counter.getCount();  // 11
// You can NEVER access 'count' directly — it's truly private!

// Two counters are fully independent — each has its own closure
const c2 = makeCounter(100);
c2.increment(); // c2's count = 101, counter's count still 11
Memoization — caching expensive results with closures
function memoize(fn) {
  const cache = new Map(); // closure — persists across all calls

  return function(...args) {
    const key = JSON.stringify(args);

    if (cache.has(key)) {
      console.log('cache hit!');
      return cache.get(key);
    }

    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}

const heavyCalc = (n) => { /* imagine 200ms CPU work */ return n * n; };
const fastCalc = memoize(heavyCalc);

fastCalc(100); // computes — 200ms. cache now has key "[100]"
fastCalc(100); // cache hit! — 0ms
fastCalc(200); // computes — 200ms (different args)
Currying — partial application using closures
// A curried function returns a new function for each argument
const multiply = a => b => a * b;
// multiply(2) returns: b => 2 * b  (2 is captured in closure)

const double = multiply(2);  // a = 2, now waiting for b
double(5);  // 10
double(21); // 42

// Real-world use: configurable/reusable functions
const addTax = rate => price => Math.round(price * (1 + rate));
const addGST  = addTax(0.18);  // rate captured in closure
const addVAT  = addTax(0.20);

products.map(p => ({ ...p, price: addGST(p.price) }));
// Clean, reusable, no magic numbers repeated everywhere
React's useState is a closure. The state value lives in React's internal scope. Every re-render creates a new closure capturing the current state value. That's why stale closures are a thing in React — if a callback captures an old state value, it won't see the updated one. The rule: always pass a function updater setState(prev => prev + 1) when new state depends on old state.
