TypeScript

TypeScript Mastery Guide

From JavaScript to type-safe code — every important concept explained with theory, code & gotchas

Why TypeScript?
TypeScript is JavaScript with a type system bolted on top. It compiles away to plain JS. But before it compiles, it catches entire classes of bugs that JavaScript silently lets through at runtime.
TypeScript is a statically typed superset of JavaScript — every valid JS file is a valid TS file. At its core, TypeScript adds a compilation step that analyses your code for type errors before it runs. This means bugs that would crash your users' browsers at 2 AM are instead caught at your keyboard at 2 PM.

The three things TS gives you that JS doesn't:
1. Static analysis — the compiler knows the shape of every value at every point.
2. Self-documenting code — type annotations are documentation that never goes out of date.
3. Fearless refactoring — rename a field and the compiler tells you every single place that breaks.
The classic JavaScript runtime bug TypeScript prevents at compile time
// ━━ JavaScript ━━ This runs. Then it crashes at runtime.
function getUser(id) {
  return { name: 'Arjun', age: 28 };
}

const user = getUser(1);
console.log(user.adress.city); // typo: 'adress' → runtime crash
// TypeError: Cannot read properties of undefined (reading 'city')
// This crashes in production. Your users see a broken screen.

// ━━ TypeScript ━━ Caught immediately in your editor.
interface User {
  name: string;
  age:  number;
}

function getUser(id: number): User {
  return { name: 'Arjun', age: 28 };
}

const user = getUser(1);
user.adress; // ❌ Compile error: Property 'adress' does not exist
// on type 'User'. Did you mean 'address'?
// Fixed before it ever ships.
❌ JavaScript
  • Bugs found at runtime (production)
  • No autocomplete for object shapes
  • Refactoring is a game of grep
  • Function params — trust the caller
  • null/undefined errors are common
✅ TypeScript
  • Bugs found at compile time (dev)
  • Full IntelliSense autocomplete
  • Refactor with confidence
  • Function contracts are enforced
  • Strict null checks built in
TypeScript is a strict mode for your brain — tsconfig.json essentials
// tsconfig.json — always enable these in new projects
{
  "compilerOptions": {
    "strict": true,          // enables all strict checks below
    // which includes:
    "noImplicitAny": true,    // no untyped variables allowed
    "strictNullChecks": true, // null/undefined are their own types
    "strictFunctionTypes": true,

    "target": "ES2020",        // output JS version
    "module": "commonjs",
    "outDir": "./dist",
    "rootDir": "./src"
  }
}
Always enable "strict": true from day one. Retrofitting strict mode onto an existing JS codebase is painful — you'll inherit 500 type errors. Start strict. It feels restrictive for the first week and then you'll never want to code without it.
Basic Types
TypeScript's type system starts with primitives you already know from JavaScript, then adds powerful tools for annotating variables, functions, arrays, and objects.
In TypeScript, every value has a type. You can annotate types explicitly, or let TypeScript infer them automatically. Type inference is one of TS's most powerful features — you don't need to annotate everything, only where inference can't figure it out.

The most important distinction to internalize: string vs String. Lowercase = primitive type (what you almost always want). Uppercase = JavaScript wrapper object class (almost never what you want). Same for number vs Number, boolean vs Boolean.
TypeExample annotationNotes
stringlet name: string = 'Arjun'Always lowercase. Uppercase String is the JS wrapper class.
numberlet age: number = 28All numbers — int, float, hex, binary, octal.
booleanlet active: boolean = trueOnly true/false. Not truthy/falsy values.
anylet x: any = 'anything'Escape hatch. Turns off type checking. Use sparingly or never.
unknownlet x: unknown = getData()Like any, but safe — must narrow before use.
neverfunction throw(): neverFunction that never returns (throws or infinite loops).
voidfunction log(): voidFunction that returns nothing useful (no return value).
null / undefinedlet x: null = nullWith strictNullChecks on, these are their own types.
tuplelet t: [string, number]Fixed-length array with known types at each index.
Type inference — let TypeScript do the heavy lifting
// TypeScript infers the type from the initial value
let name = 'Arjun';   // inferred: string — no annotation needed
let age  = 28;       // inferred: number
let done = false;    // inferred: boolean

// Only annotate when inference can't figure it out
let items: string[] = [];   // must tell TS what the array holds
let data: User | null = null; // union — can be User or null

// Arrays — two equivalent syntaxes
const nums1: number[]       = [1, 2, 3]; // preferred style
const nums2: Array<number>  = [1, 2, 3]; // generic style

// Tuples — fixed position, fixed types
const point:   [number, number]          = [10, 20];
const record:  [string, number, boolean] = ['Arjun', 28, true];
const [uname, uage] = record; // destructure with correct types!
Function typing — annotate params and return type
// Always annotate function parameters — TS can't infer those
// Return type is often inferred but annotating is explicit documentation
function add(a: number, b: number): number {
  return a + b;
}

// Optional params with ?, default values
function greet(name: string, greeting?: string): string {
  return `${greeting ?? 'Hello'}, ${name}!`;
}

// Rest params
function sum(...nums: number[]): number {
  return nums.reduce((a, b) => a + b, 0);
}

// Function type annotation (for callbacks / HOFs)
type Transformer = (input: string) => string;
const upper: Transformer = (s) => s.toUpperCase();
any vs unknown — the crucial difference
// any — turns off ALL type checking. The escape hatch.
let x: any = getFromExternalAPI();
x.nonExistentMethod(); // ✓ TypeScript says OK — runtime crash!
x.foo.bar.baz;          // ✓ TypeScript says OK — runtime crash!

// unknown — type-safe alternative to any
// You MUST narrow the type before you can use it
let y: unknown = getFromExternalAPI();
y.nonExistentMethod(); // ❌ Error: Object is of type 'unknown'

// To use it, narrow first:
if (typeof y === 'string') {
  y.toUpperCase(); // ✓ Now TS knows it's a string
}
Never use any as a default. any is a type-safety hole — it propagates and infects surrounding code. Use unknown for truly unknown external data, then narrow it. If you're using any frequently, your TypeScript is just JavaScript with extra steps.
Interfaces vs Type Aliases
Both define the shape of an object. There are real differences, and the TypeScript team has strong opinions on when to use which. Let's clear this up definitively.
interface and type are the two ways to name and reuse a type shape. They look almost identical and most of the time are interchangeable. But there are key differences:

interface: Can be extended with extends. Can be merged (declaration merging — same name, different file = merged shape). Best for defining object shapes and class contracts. This is the official TS team recommendation for objects.

type alias: Can do everything interface can, PLUS it can represent unions (A | B), intersections (A & B), primitives, tuples, and mapped types. More powerful, more flexible. Use type when you need those extra features.
Interface — the standard for object shapes
interface User {
  readonly id:   number;    // readonly — can't reassign after creation
  name:            string;
  email:           string;
  age?:            number;   // optional property
  avatar:          string | null; // can be string or null
}

// Extending — building on top of existing interfaces
interface AdminUser extends User {
  role:        'admin' | 'superadmin';
  permissions: string[];
}

// Declaration merging — unique to interface!
// If you declare the same interface name twice, TS merges them
interface Window {
  myCustomProperty: string; // adds to the global Window type
}
// This is how you extend third-party types (like Express Request)
Type alias — needed for unions, primitives, mapped types
// Union types — ONLY possible with type alias
type Status    = 'active' | 'inactive' | 'pending';
type ID        = string | number;
type Nullable<T> = T | null;

// Intersection types — combine multiple types
type AdminUser = User & { role: string; permissions: string[] };

// Discriminated union — the most powerful pattern in TypeScript
type ApiResponse<T> =
  | { status: 'success'; data: T }
  | { status: 'error';   error: string; code: number }
  | { status: 'loading' };

function handle<T>(res: ApiResponse<T>) {
  switch (res.status) {
    case 'success': return render(res.data);   // res.data is T here
    case 'error':   return showError(res.error); // res.error is string
    case 'loading': return showSpinner();
  }
}
Index signatures — for dynamic key-value shapes
// When you don't know the property names upfront
interface StringMap {
  [key: string]: string; // any string key, string value
}

interface Config {
  name:   string;     // known property
  [key: string]: unknown; // plus any other properties
}

// Record<K, V> utility type is cleaner for this (covered in Utility Types)
type Scores   = Record<string, number>;
type UserMap  = Record<string, User>;
Use interface when
Defining object/class shapes, need extends, augmenting library types, writing OOP code
Use type when
Need unions (A|B), intersections (A&B), mapped types, conditional types, or aliasing primitives
Key difference
Interfaces can be merged (declared twice). Type aliases cannot — redeclaring is an error.
The TypeScript team recommendation: use interface for objects/classes, use type for everything else. In practice, many senior devs use type for everything — it's more consistent and flexible. Pick one convention and stick to it across your codebase.
Generics
Generics let you write functions, classes, and types that work with any type — while still preserving full type safety. They're the key to reusable, type-safe code.
Imagine writing a function that returns the first element of an array. In JavaScript you'd write it once. In naive TypeScript, you'd write it for every type: getFirstString(arr: string[]), getFirstNumber(arr: number[])... forever. Generics solve this. They're like type-level variables — a placeholder for a type that gets filled in when the function is called.

The syntax <T> declares a type parameter. T is convention (Type), but you'll also see K (Key), V (Value), E (Element), TData, etc. When TypeScript calls the function, it infers what T is from the arguments — you rarely need to pass it explicitly.
Generic functions — write once, type-safe for everything
// Without generics — duplicated for each type
function getFirstString(arr: string[]): string | undefined { return arr[0]; }
function getFirstNumber(arr: number[]): number | undefined { return arr[0]; }
// This is horrible. With generics:

// T is a type variable — filled in at call time
function getFirst<T>(arr: T[]): T | undefined {
  return arr[0];
}

const s = getFirst(['a', 'b', 'c']); // T inferred as string → s: string|undefined
const n = getFirst([1, 2, 3]);        // T inferred as number → n: number|undefined
const u = getFirst([{ id: 1 }]);      // T inferred as {id:number} ← even objects!

// Multiple type parameters
function pair<A, B>(a: A, b: B): [A, B] {
  return [a, b];
}
const p = pair('name', 42); // [string, number]
Generic constraints — restricting what T can be
// T must have a .length property (string, array, etc.)
function logLength<T extends { length: number }>(value: T): T {
  console.log(value.length);
  return value;
}
logLength('hello');   // ✓ strings have .length
logLength([1, 2, 3]); // ✓ arrays have .length
logLength(42);        // ❌ Error: number has no .length

// Constrain to keys of an object — extremely useful pattern
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { name: 'Arjun', age: 28, city: 'Ahmedabad' };
getProperty(user, 'name'); // ✓ returns string
getProperty(user, 'age');  // ✓ returns number
getProperty(user, 'xyz');  // ❌ Error: 'xyz' is not a key of user
Generic interfaces and classes — the React useState pattern
// Generic interface
interface ApiResponse<TData> {
  data:    TData;
  status:  number;
  message: string;
}

// Use it with concrete types
type UserResponse  = ApiResponse<User>;
type UsersResponse = ApiResponse<User[]>;
type PostResponse  = ApiResponse<Post>;

// Generic class — like a typed container
class Stack<T> {
  private items: T[] = [];

  push(item: T): void   { this.items.push(item); }
  pop():          T | undefined { return this.items.pop(); }
  peek():         T | undefined { return this.items.at(-1); }
  isEmpty():      boolean       { return this.items.length === 0; }
}

const strStack = new Stack<string>();
strStack.push('hello'); // ✓
strStack.push(42);      // ❌ Error: number is not assignable to string
Generic constraints with extends are used everywhere in real codebases. The pattern K extends keyof T means "K must be a valid key of T" — this is the foundation of utility types like Pick, Omit, and Record. Understand it and utility types become obvious.
Type Narrowing
TypeScript tracks type possibilities through your control flow. Type narrowing is how you help it — you check a condition, and TS narrows the type from a wide possibility to a specific one.
When you have a value of type string | number, TypeScript won't let you call string methods on it because it might be a number. Narrowing is the process of refining that union to a specific type inside a conditional block. TypeScript performs control flow analysis — it tracks what you've checked and narrows the type accordingly within each branch.

There are multiple ways to narrow: typeof for primitives, instanceof for class instances, in for property existence, discriminated unions via a shared "tag" property, and custom type guards for complex cases.
typeof — narrowing primitives
function process(value: string | number | boolean) {
  if (typeof value === 'string') {
    // Inside here, TypeScript KNOWS value is string
    return value.toUpperCase(); // ✓ string methods available
  }

  if (typeof value === 'number') {
    // value narrowed to number here
    return value.toFixed(2); // ✓ number methods available
  }

  // Here TS knows value must be boolean (the only remaining option)
  return value ? 'yes' : 'no';
}
instanceof — narrowing class instances
class Cat  { meow()  { return 'meow'; } }
class Dog  { bark()  { return 'woof'; } }

function makeSound(animal: Cat | Dog) {
  if (animal instanceof Cat) {
    return animal.meow(); // ✓ animal is Cat here
  }
  return animal.bark();   // ✓ animal is Dog here
}
Discriminated unions — the most powerful narrowing pattern
// A "discriminant" is a shared property with literal types
type Shape =
  | { kind: 'circle';    radius:             number }
  | { kind: 'rectangle'; width: number; height: number }
  | { kind: 'triangle';  base:  number; height: number };

function getArea(shape: Shape): number {
  switch (shape.kind) {       // ← discriminant property
    case 'circle':
      return Math.PI * shape.radius ** 2; // shape.radius ✓
    case 'rectangle':
      return shape.width * shape.height;   // shape.width ✓
    case 'triangle':
      return (shape.base * shape.height) / 2;
  }
}
// Add a new shape variant? TS forces you to handle it in every switch.
Custom type guards — for when TS can't narrow automatically
// Return type "value is Type" is a type predicate
// When this function returns true, TS narrows the type
function isUser(value: unknown): value is User {
  return (
    typeof value === 'object' &&
    value !== null &&
    'name' in value &&
    'email' in value
  );
}

const raw: unknown = JSON.parse(apiResponse);

if (isUser(raw)) {
  // raw is narrowed to User inside this block ✓
  console.log(raw.name); // no error
}

// Exhaustiveness checking — TS tells you when you miss a case
function assertNever(x: never): never {
  throw new Error(`Unexpected value: ${x}`);
}
// Add assertNever(shape) as the default: case in a switch
// TS will error if you add a new Shape variant and forget to handle it
Discriminated unions + exhaustiveness checking is the pattern that makes TypeScript shine in large codebases. When a new team member adds a new Shape variant, every switch statement that doesn't handle it will produce a compile error. The compiler guides developers to correct completions.
Utility Types
TypeScript ships with a library of built-in generic types for common type transformations. These save you from rewriting the same mapped types everywhere. Learn these and you'll write dramatically less boilerplate.
Utility types are built-in generic types that transform other types. They're all built using mapped types and conditional types under the hood (which we cover in Advanced Types). You don't need to understand the internals to use them — just know what each does. They are globally available — no import needed.

The most important ones in daily development: Partial, Required, Readonly, Pick, Omit, Record, Exclude, Extract, ReturnType, Parameters.
Partial<T>
Makes all properties optional. Perfect for update/patch operations.
Partial<User> → all fields optional
Required<T>
Makes all properties required. Opposite of Partial.
Required<User> → no optional fields
Readonly<T>
Makes all properties read-only. Useful for immutable data.
Readonly<User> → no mutation
Pick<T, K>
Creates type with only the properties K from T.
Pick<User, 'name'|'email'>
Omit<T, K>
Creates type with all properties of T except K.
Omit<User, 'password'>
Record<K, V>
Creates type with keys K and values V.
Record<string, number>
Exclude<T, U>
Removes types from T that are assignable to U.
Exclude<string|null, null> → string
Extract<T, U>
Keeps only types from T that are assignable to U.
Extract<A|B|C, A|B> → A|B
NonNullable<T>
Removes null and undefined from T.
NonNullable<string|null> → string
ReturnType<T>
Gets the return type of a function type T.
ReturnType<typeof getUser>
Parameters<T>
Gets the parameter types of a function as a tuple.
Parameters<typeof fetch>
Awaited<T>
Unwraps Promise types. Gets the resolved type.
Awaited<Promise<User>> → User
Utility types in real API code — the patterns you'll use daily
interface User {
  id:       number;
  name:     string;
  email:    string;
  password: string;
  avatar?:  string;
}

// Partial — for PATCH/update endpoints (all fields optional)
async function updateUser(id: number, data: Partial<User>) {
  // data can have any subset of User's fields
}
updateUser(1, { name: 'New Name' });   // ✓
updateUser(1, { email: 'new@email.com', avatar: 'url' }); // ✓

// Omit — never expose password to the frontend
type PublicUser = Omit<User, 'password'>;

// Pick — just the fields you need for a specific view
type UserPreview = Pick<User, 'id' | 'name' | 'avatar'>;

// Record — a map of userId → User
type UserCache = Record<number, PublicUser>;

// ReturnType — infer the type without importing the interface
async function fetchUser(): Promise<User> { /* ... */ }
type FetchUserResult = Awaited<ReturnType<typeof fetchUser>>; // = User

// Readonly — freeze data coming from the server
function loadConfig(): Readonly<Config> {
  return { apiUrl: 'https://...', timeout: 5000 };
}
const cfg = loadConfig();
cfg.timeout = 3000; // ❌ Error: Cannot assign to readonly property
Partial<T> + Required<Pick<T, 'id'>> is the pattern for update operations — require the ID to identify the record, but make all other fields optional. This is the exact shape of a PATCH request body. You'll write this combination constantly.
Enums & Literal Types
Two ways to express a fixed set of allowed values. They look similar but behave very differently. Senior devs have strong opinions about this — and for good reason.
When you have a value that can only be one of a fixed set of options (like status: 'active' | 'inactive'), TypeScript gives you two tools: enum (a runtime construct that generates JavaScript code) and literal type unions (a compile-time only construct that compiles away completely).

The important truth: The TypeScript team, the language creator (Anders Hejlsberg), and most senior TS developers recommend avoiding regular enum in favour of literal type unions or const enum for most use cases. Enums have footguns — numeric enums in particular can cause serious bugs.
Numeric enums — the one to be careful with
// Numeric enum — generates runtime JS code, values are numbers
enum Direction {
  Up,     // = 0
  Down,   // = 1
  Left,   // = 2
  Right,  // = 3
}

Direction.Up;    // 0
Direction[0];   // 'Up' ← reverse mapping is generated

// THE BIG GOTCHA — numeric enums accept any number!
function move(dir: Direction) { /* ... */ }
move(Direction.Up); // ✓
move(0);            // ✓ TypeScript accepts this!
move(999);          // ✓ TypeScript ACCEPTS THIS TOO! Bug!
String enums — safer, but still generates JS code
// String enums — values are strings, no reverse mapping, no random numbers
enum Status {
  Active   = 'ACTIVE',
  Inactive = 'INACTIVE',
  Pending  = 'PENDING',
}
type UserStatus = Status;
setStatus(Status.Active);   // 'ACTIVE'
setStatus('ACTIVE');         // ❌ Error — must use enum member

// Generates this JS (it's a real object at runtime):
var Status;
(function(Status) {
  Status['Active']   = 'ACTIVE';
  Status['Inactive'] = 'INACTIVE';
})(Status ||= {});
Literal unions — the modern recommended approach
// ✅ Preferred by most senior TS devs — compiles to NOTHING
type Status = 'active' | 'inactive' | 'pending';
type Direction = 'up' | 'down' | 'left' | 'right';

function setStatus(s: Status) { /* ... */ }
setStatus('active');  // ✓
setStatus('ACTIVE');  // ❌ Error: Type '"ACTIVE"' is not assignable
setStatus('deleted'); // ❌ Error: Type '"deleted"' is not assignable

// Const object — namespace-like access with zero runtime overhead
const STATUS = {
  Active:   'active',
  Inactive: 'inactive',
  Pending:  'pending',
} as const; // 'as const' makes values literal types, not just string

type Status = typeof STATUS[keyof typeof STATUS];
// = 'active' | 'inactive' | 'pending'
// Usage: STATUS.Active gives you 'active' with full autocomplete
// This is the best of both worlds — enum-like API, zero JS output
'as const' — the most powerful assertion in TypeScript
// Without as const — TS widens the types
const config1 = { url: 'https://api.com', timeout: 5000 };
// type: { url: string; timeout: number } — wide types

// With as const — every value becomes its literal type
const config2 = { url: 'https://api.com', timeout: 5000 } as const;
// type: { readonly url: "https://api.com"; readonly timeout: 5000 }
// Values are narrowed to exact literals AND made readonly

// Extremely useful for array literals turned into union types
const ROUTES = ['/home', '/about', '/contact'] as const;
type Route = typeof ROUTES[number]; // '/home' | '/about' | '/contact'
Numeric enums are the biggest footgun in TypeScript. They accept any number because the reverse mapping breaks type safety. If you must use an enum (e.g., for compatibility), use string enums. Better yet — use literal type unions + as const.
Classes & OOP in TypeScript
TypeScript takes JavaScript classes and adds access modifiers, abstract classes, decorators, and parameter properties — making OOP patterns from Java/C# feel natural.
TypeScript classes compile to regular JavaScript classes but add compile-time access control and type checking. The key additions: public (default — accessible everywhere), private (only within the class), protected (within the class and subclasses), readonly (can only be assigned in constructor).

TypeScript also supports abstract classes — classes you can't instantiate directly, only extend. They define a contract that subclasses must implement. This is the TypeScript way to define base classes with enforced method signatures.
Access modifiers — TypeScript OOP fundamentals
class BankAccount {
  // public: accessible everywhere (this is the default)
  public owner: string;

  // readonly: set in constructor, never changed after
  readonly accountNumber: string;

  // private: only accessible within BankAccount class
  private balance: number;

  // protected: accessible within this class AND subclasses
  protected transactionLog: string[] = [];

  constructor(owner: string, initialBalance: number) {
    this.owner         = owner;
    this.balance        = initialBalance;
    this.accountNumber  = generateId();
  }

  deposit(amount: number): void {
    if (amount <= 0) throw new Error('Amount must be positive');
    this.balance += amount;
    this.transactionLog.push(`+${amount}`);
  }

  // Getter — accessed like a property, not a method
  get currentBalance(): number { return this.balance; }
}

const account = new BankAccount('Arjun', 1000);
account.deposit(500);
account.balance;       // ❌ Error: private property
account.currentBalance; // ✓ 1500 — accessed via getter
Parameter properties — TypeScript shorthand (used everywhere)
// Without shorthand — verbose and repetitive
class ServiceLong {
  private db: Database;
  private logger: Logger;
  constructor(db: Database, logger: Logger) {
    this.db = db; this.logger = logger;
  }
}

// With parameter properties — much cleaner
class Service {
  constructor(
    private readonly db:     Database,  // declares AND assigns in one step
    private          logger: Logger,
    public           name:   string,
  ) {} // constructor body can be empty!
}
Abstract classes — define contracts for subclasses
// Abstract class — can't be instantiated, only extended
abstract class Animal {
  constructor(protected name: string) {}

  // Abstract method — subclasses MUST implement this
  abstract makeSound(): string;

  // Concrete method — subclasses inherit this
  toString(): string {
    return `${this.name} says: ${this.makeSound()}`;
  }
}

class Dog extends Animal {
  makeSound() { return 'Woof!'; } // ✓ must implement
}

class Cat extends Animal {
  // makeSound() missing → ❌ Error: must implement abstract method
}

new Animal('?'); // ❌ Error: Cannot create instance of abstract class
new Dog('Rex').toString(); // ✓ "Rex says: Woof!"
Use parameter properties (constructor shortcuts) in every TypeScript class. It's not a shortcut — it's idiomatic TypeScript. Every framework (NestJS, Angular) uses it heavily. You declare, type, and assign a property in a single line. Writing it the long way is just boilerplate.
Advanced Types
Mapped types, conditional types, template literal types, infer — these are the building blocks TypeScript uses to build its own utility types. Understanding these unlocks the full power of the type system.
Advanced types let you compute types from other types at compile time. Think of them as type-level functions — they take types as inputs and produce new types as output. This is what makes TypeScript's type system truly powerful and what separates a TypeScript expert from a beginner who just adds : string everywhere.

Mapped types transform all properties of a type. Conditional types select a type based on a condition. Template literal types build string union types from combinations. The infer keyword extracts a type from within a conditional type.
Mapped types — transform every property of a type
// Mapped type syntax: [K in keyof T]: ...
// This is how Partial, Readonly, Required are implemented

// Recreating Partial from scratch to understand the pattern
type MyPartial<T> = {
  [K in keyof T]?: T[K]; // ? makes optional, T[K] is the original type
};

// Recreating Readonly
type MyReadonly<T> = {
  readonly [K in keyof T]: T[K];
};

// Custom: nullable all values
type Nullable<T> = { [K in keyof T]: T[K] | null };

// Custom: make specific keys required, rest partial
type WithRequired<T, K extends keyof T> =
  Partial<T> & Required<Pick<T, K>>;

// Create an update type: id required, rest optional
type UpdateUser = WithRequired<User, 'id'>;
// { id: number } & Partial<User> → id required, rest optional
Conditional types — type-level if/else
// Conditional type: T extends U ? TypeIfTrue : TypeIfFalse
type IsString<T> = T extends string ? 'yes' : 'no';
type A = IsString<string>; // 'yes'
type B = IsString<number>; // 'no'

// 'infer' — extract a type from within a conditional
// This is how ReturnType is implemented
type MyReturnType<T> =
  T extends (...args: any[]) => infer R
    ? R   // R is inferred as the return type
    : never;

type X = MyReturnType<() => string>; // string
type Y = MyReturnType<(n: number) => User>; // User

// Extract element type from array
type ElementOf<T> = T extends (infer E)[] ? E : never;

// Build string types from combinations
type EventName = string;
type EventHandler<T extends string> = `on${Capitalize<T>}`;

type ClickHandler = EventHandler<'click'>;  // 'onClick'
type HoverHandler = EventHandler<'hover'>;  // 'onHover'

// Generate getter names from object keys
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
};

type UserGetters = Getters<{ name: string; age: number }>;

keyof, typeof, infer, and mapped types are the four pillars of advanced TypeScript. Once you understand that TypeScript's own utility types are just combinations of these four concepts, you can write any custom utility type you need. Start by reading the TypeScript source for Partial, Required, and ReturnType — it's illuminating.
TypeScript with React
Where you'll use TypeScript most in the real world. Typing props, hooks, events, refs, and context — everything you need to write type-safe React components.
TypeScript + React is the most common pairing in the ecosystem. The key is understanding a few React-specific generic types: React.FC (function component), React.ReactNode (anything renderable), React.ComponentProps (extract props from a component), event types like React.ChangeEvent and React.MouseEvent, and the useRef generic for DOM element references.

One important note: The React team no longer recommends React.FC for defining components. Define component functions with plain function syntax and type the props argument directly — it's simpler, and TypeScript infers the return type automatically.
Typing React component props — the right way
// Define props with interface (preferred for objects)
interface ButtonProps {
  label:      string;
  onClick:    () => void;
  variant?:   'primary' | 'secondary' | 'danger';
  disabled?:  boolean;
  children?:  React.ReactNode; // anything React can render
}

// ✅ Preferred: plain function, typed prop argument
function Button({ label, onClick, variant = 'primary', disabled = false }: ButtonProps) {
  return (
    <button
      onClick={onClick}
      disabled={disabled}
      className={`btn btn-${variant}`}
    >
      {label}
    </button>
  );
}

// ❌ Not recommended anymore: React.FC adds implicit children and other quirks
const Button: React.FC<ButtonProps> = ({ label }) => <button>{label}</button>;
Typing useState — when inference isn't enough
// TypeScript infers from initial value — no annotation needed
const [count, setCount]   = useState(0);        // number
const [name,  setName]    = useState('');        // string
const [active, setActive] = useState(false);     // boolean

// Annotate when initial value is null or an empty array
const [user, setUser]     = useState<User | null>(null);
const [users, setUsers]   = useState<User[]>([]);

// Using a discriminated union for complex state
type FetchState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error';   error: string };

const [state, setState] = useState<FetchState<User>>({ status: 'idle' });

if (state.status === 'success') {
  state.data; // ✓ narrowed to User — no other status has .data
}
Typing events and useRef
// Event types — always use React.XEvent over native Event
function Form() {
  // Input change event — HTMLInputElement is the target type
  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    console.log(e.target.value);      // ✓ string
    console.log(e.target.checked);    // ✓ boolean (for checkboxes)
  };

  // Form submit event
  const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault(); // ✓
  };

  // Button click — MouseEvent
  const handleClick = (e: React.MouseEvent<HTMLButtonElement>) => {
    console.log(e.currentTarget.textContent);
  };

  // useRef — always provide the element type
  const inputRef = useRef<HTMLInputElement>(null);
  // inputRef.current is HTMLInputElement | null until it mounts
  const focusInput = () => inputRef.current?.focus(); // optional chaining!

  return <input ref={inputRef} onChange={handleChange} />;
}
Typing useContext — the complete pattern
// 1. Define the context shape
interface AuthContextType {
  user:   User | null;
  login:  (creds: Credentials) => Promise<void>;
  logout: () => void;
}

// 2. Create context with a default or undefined
const AuthContext = createContext<AuthContextType | undefined>(undefined);

// 3. Create a typed hook that ensures context is used inside provider
function useAuth(): AuthContextType {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth must be inside AuthProvider');
  return ctx; // narrowed — no longer AuthContextType | undefined
}

// 4. Use in any component — fully typed
function Profile() {
  const { user, logout } = useAuth();
  return <div>{user?.name}</div>;
}

React.ComponentProps<typeof MyComponent> gives you a custom component's props. This means you can extend them without rewriting the full interface.
